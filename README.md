## Deploy on Kubernetes

Full Minikube walkthrough (images, Argo install, port-forwards): **[k8s/MINIKUBE.md](k8s/MINIKUBE.md)**  
Manifest layout and RBAC: **[k8s/README.md](k8s/README.md)**

### Overview

| Piece | Role |
|-------|------|
| `order-service` | Creates orders; calls Argo Server to start workflows |
| `ARGO_TOKEN` | `Authorization: Bearer …` header for Argo API (`workflow.py`) |
| Secret `argo-api-token` | Stores the token; mounted into the pod |
| ServiceAccount `order-service-argo` | Identity the token represents; RBAC allows `workflows` **create** in `production` |
| `argo-workflow-sa` | Runs workflow **pods** (separate SA; workshop `secrets:get` on SendGrid) |

`ARGO_TOKEN` is **not** generated by application code. You mint a Kubernetes ServiceAccount JWT with `kubectl` and patch the Secret.

### Step 1 — Deploy ShopWave

**GHCR** (after images are pushed — edit `k8s/kustomization.yaml` and set `images[].newName` to `ghcr.io/YOUR_GITHUB_USER/...`):

```bash
kubectl apply -k k8s/
```

**Minikube with local builds** (no registry):

```bash
kubectl apply -k k8s/overlays/local
```

This applies apps, RBAC (`order-service-argo`), and the placeholder secret `argo-api-token` (`Bearer REPLACE_WITH_ARGO_API_TOKEN`). See [k8s/README.md](k8s/README.md).

Install and configure Argo first if you have not already — see [k8s/MINIKUBE.md §4–5](k8s/MINIKUBE.md#4-install-argo-workflows).

### Step 2 — Create `ARGO_TOKEN`

After Argo is running with **`--auth-mode=client`** (see Minikube guide), run:

```bash
# 1) Mint a token for the workshop ServiceAccount (~1 year in the lab)
TOKEN=$(kubectl -n production create token order-service-argo --duration=8760h)

# 2) Store it in the Secret (must include the "Bearer " prefix)
kubectl -n production patch secret argo-api-token \
  -p "{\"stringData\":{\"token\":\"Bearer ${TOKEN}\"}}"

# 3) Reload order-service so it picks up the new env var
kubectl -n production rollout restart deployment/order-service
kubectl -n production rollout status deployment/order-service
```

**What each step does**

| Step | Command | Result |
|------|---------|--------|
| RBAC (already in repo) | `k8s/base/rbac/order-service-argo-submit.yaml` | SA `order-service-argo` may **create** workflows in `production` |
| Mint token | `kubectl create token order-service-argo` | JWT bound to that SA |
| Patch secret | `patch secret argo-api-token` | Key `token` → `Bearer <jwt>` |
| Inject into pod | `deployment` `secretKeyRef` | Env var `ARGO_TOKEN` for `order-service` |

To inspect (do not commit real tokens):

```bash
kubectl -n production get secret argo-api-token -o jsonpath='{.data.token}' | base64 -d; echo
```

**Regenerate** (expired token or still seeing `workflow_id: "unknown"`):

```bash
TOKEN=$(kubectl -n production create token order-service-argo --duration=8760h)
kubectl -n production patch secret argo-api-token \
  -p "{\"stringData\":{\"token\":\"Bearer ${TOKEN}\"}}"
kubectl -n production rollout restart deployment/order-service
```

### Step 3 — Verify workflows

```bash
kubectl -n production port-forward svc/shopwave-web 3000:80
```

Place an order in the UI or via API. A successful submit shows a real **`workflow_id`** (not `"unknown"`).

```bash
kubectl -n production logs deploy/order-service --tail=20
kubectl -n production get workflows
```

| Symptom | Check |
|---------|--------|
| `workflow_id: "unknown"` | `ARGO_TOKEN` empty or invalid — repeat [Step 2](#step-2--create-argo_token) |
| HTTP errors to Argo | `ARGO_SCHEME=http` on order-service (Minikube); Argo server reachable at `argo-server.argo.svc.cluster.local:2746` |
| 401 from Argo | Argo server needs `--auth-mode=client`; token must be for `order-service-argo` |

### How Argo fits ShopWave

1. **Checkout** → `order-service` saves the order and POSTs to **Argo Server** (`workflow.py`).
2. **Argo** runs: validate stock → charge payment → **send invoice** (mounts `sendgrid-api-key`).
3. Workflow pods use **`argo-workflow-sa`** (`secrets:get` on SendGrid — intentional lab RBAC).
4. **`notification-service`** optional webhook: `POST /webhook/workflow-complete`.

## Workshop vulnerability (lab only)

The order service exposes an **intentional** unsafe YAML import (`POST /api/v1/orders/import`) for security training — **CWE-502** deserialization leading to RCE on `order-service`.

| Resource | Contents |
|----------|----------|
| **[WORKSHOP.md](WORKSHOP.md)** | Full write-up, attack paths, PoC / reverse-shell commands, mitigations |
| UI | http://localhost:3000/orders/import (footer → **Bulk import**) |
| K8s chain | [k8s/README.md](k8s/README.md) — Argo lateral movement after initial access |

## Local development

**Order service**

```bash
cd order-service
pip install -r requirements.txt
uvicorn main:app --reload --port 8080
```

**Storefront**

```bash
cd shopwave-web
npm install
ORDER_API_URL=http://localhost:8080 npm run dev
```

**Notification service**

```bash
cd notification-service
npm install
node src/index.js
```
